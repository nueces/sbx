Small boxes for dungeon runs.

sbx sends your coding agent into a disposable VM, so you can explore risky code safely without handing the dungeon your host credentials.

Start a run Read the safety rules

The quest: Explore risky code safely.

Coding agents are powerful, but unknown repositories are dungeons. Build scripts, package installs, shell commands, and generated patches can all hide traps. sbx lets the agent explore inside a small box instead of your host machine.

▣

Disposable VM

Create, enter, inspect, and throw away a QEMU-backed SmolVM sandbox when the run is done.

Agent-ready

Run Pi, Claude, or Codex inside the guest while keeping your host environment out of reach by default.

Credential cautious

Host credentials are not copied unless you explicitly ask for them. Safe Git identity forwarding is separate.

The dungeon loop

Pick a box

Use a preset image or a local ready-to-run image for your agent.

Enter the run

sbx run creates or starts the VM, exposes auth callback forwarding, and attaches to the agent.

Explore

Mount your project, run commands, test patches, and keep risky effects inside the guest.

Leave clean

Stop, recreate, or destroy the sandbox when the dungeon has been mapped.

Summoning steps / Installation

# install the released tool and check the defaults
$ uv tool install git+https://github.com/nueces/sbx.git@v0.2.3
$ sbx doctor

# select your quest and jump into it
$ cd into/your/project

# create if missing, start if stopped, attach to your Agent
$ sbx run the-quest

# open a shell in an existing box
$ sbx shell the-quest

# throw away the dungeon state
$ sbx recreate the-quest --force

What the small box carries

Core commands

  • sbx run [NAME] — main agent workflow
  • sbx create [NAME] — provision without attaching
  • sbx shell [NAME] — open a guest shell
  • sbx recreate [NAME] — delete and start fresh
  • sbx network status [NAME] — inspect tunnels
  • sbx completion SHELL — generate shell completions
  • sbx image ls — list local ready-to-run images
  • sbx image build-debian — build a local Debian/Pi image

Treasure maps for repeatable runs

Put the route in .sbx.toml — or let the first sbx run/sbx create bootstrap a minimal file for a new VM: name, agent, and selected options. The next dungeon run starts from the same map, so teammates and future-you can reproduce the same sandbox path. 

[sbx]
agent = "pi"
name = "project-sbx"
project_path = "."
mount = ["/home/me/code/tooling"]
writable_mounts = true
run_user = "agent"
copy_host_credentials = false
git_config = true

Forge reusable images

Build a local Debian/Pi image once with sbx image build-debian, then reuse it in your dungeon maps. Cast the build spell with --with-docker when the run needs extra power, like rootless Docker.

Ergonomics: the easy path through the dungeon

A good dungeon tool should be easy to reach for. sbx keeps the command loop small and memorable: run to enter, shell to inspect, recreate to reset, and rm when the quest is over. The safe route becomes muscle memory instead of another ritual to remember.

Safety rules

Protected by default

  • Agents run in the VM, not on the host.
  • Host credential files are not copied by default.
  • Project mounts can be explicit and disposable.
  • OAuth callback forwarding is tracked by sbx.

Know the trap doors

sbx is isolation, not magic. Only copy host credentials when you mean it. Treat writable mounts as shared ground between the adventurer and the dungeon.

Coming soon: package auth without handing over the treasure

Package-auth mediation is in progress. The goal is to let tools like uv sync reach authenticated package indexes from inside the VM while real package tokens stay on the host.

Not public-ready yet

This feature is still being hardened. When it lands, the intended flow is: the guest receives a placeholder credential, and a host-side proxy swaps it only for the configured package path.
